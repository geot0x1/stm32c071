## Project Overview

**STM32C071 PWM Repeater and Governor** — A firmware project for a dual-channel PWM signal repeater with temperature-based fan control. The system reads two independent PWM input signals (120-200 Hz), measures their duty cycles, applies software-defined throttle limits, and generates two independent 160 Hz PWM outputs. It includes thermal monitoring via DS18B20 and internal sensors, with fan control logic using temperature hysteresis.

## Build & Development Commands

### Prerequisites
- **CMake** ≥ 3.22
- **Ninja** build system
- **gcc-arm-none-eabi** ARM cross-compiler
- **STM32CubeProgrammer** CLI for flashing
- **Python 3** (for build and flash scripts)

### Build

```bash
# Full build (configure + compile)
python build.py

# Clean rebuild
python build.py --clean
```

The build system uses CMake with Ninja generator. Build output goes to `./build/` directory. Generated binary is recorded in `build/build_info.json`.

### Flash & Run

```bash
# Build and flash (run.py calls both build.py and flash.py)
python run.py

# Flash only (requires prior build)
python flash.py
```

Flashing uses SWD (Serial Wire Debug) interface via `STM32_Programmer_CLI` with automatic device reset (`-rst` flag).

### Serial Monitoring

```bash
python monitor_com4.py
```

Streams serial output from COM4 (default). Modify the script for different COM ports or baud rates.

## Architecture & Module Structure

### Core Project Layout

- **`Core/Src/`** — Main application code
  - `main.c` — Entry point, peripheral initialization, main event loop
  - `pwm_repeater.c` — PWM input capture and output generation logic
  - HAL/initialization files (generated by STM32CubeMX)

- **`Core/Inc/`** — Header files for core modules

- **Hardware Peripheral Files**
  - `cmake/stm32cubemx/` — STM32CubeMX-generated peripheral drivers and CMSIS
  - `STM32C071XX_FLASH.ld` — Linker script for flash memory layout
  - `startup_stm32c071xx.s` — ARM assembly startup code

### Library Modules (CMake subdirectories)

Each module is a self-contained library linked into the final executable. Key modules:

- **`pwm_repeater.c`** — Core PWM signal processing (input capture, throttle logic, output generation)
  - Uses TIM2 (32-bit timer) for input capture (PB10, PB11) and PWM output (PA0, PA1)
  - Implements sliding CCR approach for phase-independent output
  - Critical: No mid-cycle timer resets; only use ARPE and preload safety mechanisms

- **`temperature_sensor/`** — Unified temperature interface
  - Aggregates external DS18B20 (1-Wire, PB3) and internal sensor readings
  - Returns `max(external, internal)` as system temperature

- **`ds18b20/`** — DS18B20 digital temperature sensor driver
  - 1-Wire protocol implementation

- **`onewire/`** — 1-Wire bus protocol

- **`fan_control/`** — Fan control logic
  - 2-wire (ON/OFF) and 3/4-wire (PWM) fan support
  - Hysteresis-based control (upper threshold turns ON, lower threshold turns OFF)
  - DIP switch configuration for fan type selection

- **`delay/`** — Delay/timing utilities

- **`sys_time/`** — System time and tick management

- **`critical/`** — Critical section management (interrupt disable/enable)

- **`nvs/`** — Non-Volatile Storage (EEPROM/Flash)

- **`crc32/`** — CRC32 checksum utilities

- **`usb/`** — TinyUSB integration for CDC (serial over USB)

## Key Design Constraints

### PWM Processing

- **Input Frequency Range**: 40–500 Hz (system rejects frequencies outside this range)
- **Valid Operational Range**: 120–200 Hz (synchronized with display dimming)
- **Output Frequency**: Fixed at 160 Hz (compile-time constant)
- **Throttle Logic**: `output_dc = min(input_dc, throttle_limit)` per channel
- **Startup Handling**: System must gracefully handle hot-swap scenarios where input cables are plugged/unplugged
- **Noise Filtering**: Input validation requires consecutive, mathematically valid high/low level pairs; single edges or noise-induced transitions reset the validation state machine

### Critical Implementation Rules (DO NOT VIOLATE)

1. **No Mid-Cycle Resets** — Never use `TIM_EGR_UG` (Update Generation) inside ISRs to force register updates. This causes severe output phase jitter. Use only Auto-Reload Preload (`ARPE`) and Output Compare Preload for safe register swaps at cycle boundaries.

2. **64-bit Scaling Math** — All CCR scaling calculations must use 64-bit integer arithmetic to prevent overflow. Always scale against the *hardware's current ARR*, not just the measured input period, to prevent flickering during frequency transients.

3. **Level-Based Validation** — Only accept periods that result from consecutive, mathematically valid high and low level measurements. Reject single edges or "ghost" levels caused by electrical noise.

### Temperature & Fan Control

- **Hysteresis**: Upper threshold triggers all fans ON; lower threshold triggers all fans OFF. Prevents rapid on/off cycling (flickering).
- **Temperature**: System uses `max(external_sensor, internal_sensor)` to determine current thermal state.
- **Fan Types**:
  - 2-wire: Simple ON/OFF control
  - 3-wire: ON/OFF with tachometer feedback (not required for basic operation)
  - 4-wire: PWM control (0% or 100% logic)

### DIP Switch Configuration

| Switch State | Fan Type |
|---|---|
| `00` | 2-wire FAN (ON/OFF power control) |
| `01` | 3-wire FAN (ON/OFF + tachometer) |
| `10` | 4-wire FAN (PWM 0%/100%) |

> **Note (4-wire):** Tacho reading is not mandatory. A future option is to use 4 switches to select Power PWM vs. Remote PWM for 4 independent fan channels.

### Status LED Behavior

| LED Pattern | System State |
|-------------|--------------|
| ON 100ms / OFF 2000ms | Fans are **OFF** |
| ON 100ms / OFF 1000ms | Fans are **ON** |
| ON 200ms / OFF 200ms | **ERROR** (Overheat / Sensor Lost) |

## Code Style

Follow the style guide in `.agents/rules/CODING_STYLE.md`:

- **Functions**: `snake_case` — `read_sensor()`
- **Types** (struct/enum/typedef): `PascalCase` — `SensorConfig`

- **Static/local variables**: `snake_case` — `is_ready`
- **Constants/Macros**: `UPPER_SNAKE_CASE` — `MAX_ADC`
- **Formatting**: Allman style braces (new line), 4-space indentation, no tabs, braces required for all control statements

## Common Workflows

### Modifying PWM Logic

PWM repeater code lives in `Core/Src/pwm_repeater.c` and its header `Core/Inc/pwm_repeater.h`. The implementation uses TIM2's Input Capture (IC) on channels 3 and 4 to measure incoming PWM duty cycles, and Output Compare (OC) on channels 1 and 2 to generate outputs. Changes here directly impact frequency lock and throttle behavior; always verify against the design constraints above and test with real PWM inputs.

### Adding New Sensors or Peripherals

1. Create a new subdirectory in the project root (e.g., `new_sensor/`)
2. Add a `CMakeLists.txt` to define the library, sources, and include paths
3. Create `.h` and `.c` files following the coding style
4. Update the main `CMakeLists.txt` to `add_subdirectory(new_sensor)` and link it to the executable
5. Include the header in `Core/Src/main.c` or the relevant driver file

### Testing During Development

- Use `python monitor_com4.py` to stream debug output via USB CDC serial
- Add debug printf statements to `main.c` via USB/UART (TinyUSB provides CDC serial output)
- Verify PWM input capture and output generation with an oscilloscope on PA0, PA1 (outputs) and PB10, PB11 (inputs)
- Temperature readings and fan state changes can be monitored through serial debug output
